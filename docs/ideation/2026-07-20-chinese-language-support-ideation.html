背景上下文

项目：SkillHub — 一个轻量级的、自托管的 Hermes Agent 技能注册中心

技术栈：Python 3.10+, FastAPI 后端, 静态 HTML/CSS/JS 前端

当前状态：所有 UI 文本均为硬编码英文（约30个Web UI字符串，约10个CLI字符串），无i18n库或翻译文件，<html lang="en">硬编码。

关键发现：

  • Web UI：单页应用，搜索、分类过滤、排序、技能列表（卡片）、技能详情弹窗
  • 字体栈缺少显式 CJK 回退（Windows/Linux 上中文可能渲染不一致）
  • 日期格式使用 toLocaleDateString() 但未指定中文区域
  • CLI（Click）：搜索、列表、推送、安装命令均为英文
  • 代码库小（59行HTML，180行JS），i18n集成简单直接

范围约束：只做 Web UI 的中文化，CLI 不在范围内。

主题轴

  1. UI文本提取与翻译 — 提取所有硬编码英文字符串并创建中文翻译
  2. 语言切换UX — 语言选择器、浏览器持久化、默认检测
  3. CJK文本渲染 — 字体栈、排版、中文字符布局
  4. 区域格式化 — 中文区域的日期、数字、时间格式
  5. CLI国际化 — Click命令帮助文本、错误消息、输出格式中文化 （超出范围 — 只做Web UI）

排序构思

  • 1. 硬编码英文字符串无处不在 UI文本提取与翻译
  • 2. 字体栈缺少显式CJK回退 CJK文本渲染
  • 3. 从浏览器自动检测区域设置 语言切换UX
  • 4. 单一JSON翻译文件架构 UI文本提取与翻译
  • 5. 中文优先，零英文默认 语言切换UX
  • 6. 同时翻译技能元数据 UI文本提取与翻译

1. 硬编码英文字符串无处不在

Web UI中的每个用户可见字符串（index.html + app.js）都是硬编码英文 — 搜索占位符、按钮标签、过滤器选项、加载状态、错误消息、元数据标签。中文用户看到"Search skills..."和"Loading skills..."却无法理解。

2. 字体栈缺少显式CJK回退

当前字体栈（-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif）在macOS上正常工作（PingFang SC是系统回退），但在Windows和Linux上，中文字符可能使用不一致或难以辨认的回退字体渲染。

3. 从浏览器自动检测区域设置

不要构建语言切换器UI，而是自动检测浏览器的Accept-Language头部并提供适当的区域设置。中文用户的浏览器已经表明了他们的偏好。FastAPI可以原生读取Accept-Language头部。

4. 单一JSON翻译文件架构

不要将翻译分散在HTML和JS文件中，而是将所有字符串提取到一个 locales/zh-CN.json 文件中，将字符串键映射到中文值。JS在运行时从此文件读取。对于约30个字符串，一个文件完全够用。

5. 中文优先，零英文默认

将"中文作为默认"的要求推向极致：从默认UI中移除所有英文文本。英文成为需要明确选择加入的特殊情况，而不是中文覆盖的默认值。这迫使默认语言和辅助语言之间有干净的分离。

重要性

6. 同时翻译技能元数据

将翻译系统扩展到UI框架之外，涵盖技能描述、分类和标签。技能作者可以提供其技能元数据的中文翻译。如果UI是中文的，纯英文的技能描述会创造一种不协调的混合语言体验。
